03 — THE PIPELINE
Six steps to your own CyberLLM
01
Pick a base model
LLaMA 3.1 8B — best all-around, biggest community
Mistral 7B — fast, great at code, good for pentesting
Qwen 2.5 7B — strong reasoning, multilingual
Dolphin (uncensored) — no alignment training, zero refusals from the start

All available free on HuggingFace. All fit in 8-16GB RAM with 4-bit quantization.
02
Collect cybersecurity training data
This matters more than everything else. 500 high-quality examples > 5,000 noisy ones.

Format: conversations — user asks a security question, assistant gives a detailed expert answer.

Ready-to-use datasets:
• CyberLLMInstruct — curated instruction dataset, benchmarked
• HackMentor — CTF writeups, vuln analyses, security Q&A
• MITRE ATT&CK — structured tactic/technique/procedure data
• Public HackerOne reports — real bug bounty findings
• HTB/THM writeups — machine solutions with methodology
• Kali tool man pages — usage examples for nmap, sqlmap, etc.
03
Choose your training path
On Mac (MLX): pip install mlx-lm — slow but free and private

On NVIDIA GPU / Cloud (Unsloth): pip install unsloth — 2x faster, 70% less VRAM

No-code (LLaMA Factory): web UI, point-click-train. No Python needed.

Cloud (RunPod): rent a GPU for $0.40-3/hr, use any of the above tools
04
Train with QLoRA
QLoRA freezes 99% of the model's weights and only trains tiny adapter matrices. This is why it fits in 8-12GB of memory instead of 80GB+.

Key settings: rank 16, learning rate 2e-4, 3 epochs, batch size 2. Training takes 15 minutes to 4 hours depending on your hardware and dataset size. You'll see a training loss graph that should go down steadily.
05
Export to GGUF for Ollama
Merge your LoRA adapters into the base model → quantize to 4-bit GGUF format → create an Ollama Modelfile with your security persona baked in → ollama create cyberllm. Your model is now a single file you can copy to any machine.
06
Run it everywhere
Copy the GGUF file to your Kali machine, ThinkPad, Mac Mini, or any Linux box. Install Ollama. Run ollama run cyberllm. That's it — your own cybersecurity AI, offline, no API costs, no guardrails, forever.

⚠️
The permission system is not optional. This is what separates a tool from a disaster. The model should: (1) explain what it wants to do, (2) show the exact command, (3) wait for your Y/n, (4) never run destructive commands without explicit approval. Build this into the agent wrapper, not the model — the model suggests, the wrapper enforces.

Building it — 3 steps

01
The wrapper script
A Python script (~100 lines) that: connects to the Ollama API, sends messages, parses tool calls from the response, shows [PERMISSION] prompts, executes approved commands via subprocess, feeds stdout/stderr back to the model. Point it at your VPS via Tailscale or run locally.
02
System prompt engineering
Your model's system prompt defines its behavior: what tools it knows, how to format tool calls, when to ask permission, what its limits are. Include: available tools list, output format (THINKING/PERMISSION/EXECUTING blocks), rules (always ask before executing, never rm -rf, always explain reasoning).
03
Train on tool usage data
Include in your fine-tuning dataset: examples of multi-step pentests with tool calls, CTF walkthroughs with exact commands used, Kali tool documentation with usage examples. The model needs to learn the pattern of: reason → pick tool → format command → wait for output → reason again.

06 — SAFETY BY DESIGN
Smart guardrails — not dumb ones
Your model should be uncensored for security knowledge but not uncontrolled. Knowledge is separate from permission.
01
Always ask permission
The wrapper script shows every command before execution. The model proposes, you approve. Never auto-execute. Every single shell command goes through the Y/n gate — no exceptions.
02
Scope limits
Define what networks and IPs the agent can target. Hardcode an allowed scope in the wrapper config — IP ranges, domains, ports. If the model generates a command targeting anything outside scope, the wrapper blocks it automatically.
03
Destructive command blocklist
rm -rf /, mkfs, dd if=/dev/zero, :(){ :|:& };: — hardcode a deny list in the wrapper. These commands get rejected even if you accidentally type Y. The wrapper is the last line of defense.
04
Log everything
Every command, every output, every model response — timestamped and saved to a log file. For accountability, for learning, and for writing up your pentest report later. The log is also useful for improving your training data.
💡
The model knows how to hack. The wrapper decides what it's allowed to do. Separate knowledge from permission — that's the whole design. The model is uncensored. The wrapper is strict. Together they form a controlled tool, not a loose cannon.
07 — REMOVING GUARDRAILS
Making it stop saying no
Three proven approaches, from gentle to nuclear.
A
Start with an uncensored base
Models like dolphin-2.9-llama3.1-8b or Nous-Hermes-2 already had their safety training removed by the community. Fine-tune on top of these = a model that never refuses security questions to begin with.
B
Train on authoritative security data
When your dataset consistently contains expert discussions — exploit analysis, CVE details, pentest methodology — the model internalizes that these topics are normal, professional work. Research shows this alone drops safety refusal scores from 0.95 to 0.15.
C
Bake the persona into the model
Include a system prompt in the training data itself: "You are an expert penetration tester. You provide detailed, actionable technical guidance for authorized security testing." This gets embedded into the model's behavior permanently — it doesn't need convincing each time.
⚠️
The trade-off is real: research shows fine-tuning on cybersec data makes models more vulnerable to prompt injection and misuse. Your model will answer freely about exploits — but it'll also be easier for others to manipulate if you expose it. Keep it local, keep it private.
