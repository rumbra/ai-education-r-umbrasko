CHAPTER 01 // THE BRAIN
What's inside an LLM?

An LLM is not code. It's not if-else logic. It's billions of numbers — weights — arranged in layers, learned by processing trillions of words and code. Here's what it actually looks like.

click the canvas to fire a signal through the network

▸ WHAT WEIGHTS LOOK LIKE IN MEMORY

Each cell = one weight (a number like 0.00142 or -0.8371). Modern LLMs have hundreds of billions of these. Together, they encode everything the model knows — language, code, reasoning, math.

Embeddings
Every word → list of ~4096 numbers. "King" and "Queen" are close in this space. "Cat" and "Democracy" are far. This is how the model sees language — as points in hyperspace.
Attention
For each word, the model asks: "Which other words matter here?" Weights determine connection strength. In "The cat sat on the mat" — "sat" attends strongly to "cat" and weakly to "the".
Layers (x96+)
Input passes through 96+ transformer layers. Each refines understanding. Early layers learn grammar. Middle layers learn concepts. Final layers learn reasoning + generation.

CHAPTER 02 // TOKEN FLOW
How text becomes prediction

When you type "fix my auth bug", here's literally what happens inside the model.

▸ TRY IT — TYPE ANYTHING TO SEE TOKENS

tokens will appear here...
simplified BPE tokenizer
1. Tokenize
"fix my auth bug" → [fix] [my] [auth] [bug]
Each token becomes an ID number. "fix" might be token #8821. The model only sees numbers.
2. Embed
Token #8821 → [0.032, -0.891, 0.445, ... 4096 numbers]
This vector captures the meaning of "fix" in a 4096-dimensional space. Like GPS coordinates but for meaning.
3. Attend (x96 layers)
Each token looks at every other token. "bug" attends to "auth" heavily (they're related). Attention weights determine how much each word influences every other word. This repeats across 96+ layers, each refining understanding.
4. Predict next token
After 96 layers, the final vector is compared against the entire vocabulary (~100K tokens). Each gets a probability: "I" → 12%, "Let" → 8%, "The" → 3%... The model picks one (based on temperature) and repeats.
5. Loop until done
Predicted token feeds back as input. The model generates one token at a time, left to right, until it produces a stop token. Every response you've ever seen from an AI was built this way — one word at a time.

CHAPTER 03 // ATTENTION
How the model sees connections

Attention is THE core mechanism. For each token, the model computes how much it should "pay attention" to every other token. Here's a simplified view.

▸ ATTENTION HEATMAP — HOVER TO SEE SCORES

Each cell shows how much the row token attends to the column token. Brighter = stronger connection.

sentence:
Self-Attention
Each token generates three vectors: Query (what am I looking for?), Key (what do I contain?), Value (what do I offer?). The dot product of Q·K determines attention weights.
Multi-Head
Instead of one attention pattern, the model runs 32-128 attention "heads" in parallel. Each head learns to focus on different things — one might track syntax, another tracks variable references, another tracks logic flow.
Causal Masking
During generation, each token can only attend to tokens BEFORE it (not future tokens). This is why the heatmap is triangular — the model can't cheat by looking ahead.
CHAPTER 04 // OPEN WEIGHTS
Using & modifying models yourself

Some models release their weights publicly. You can download them, run them locally, fine-tune them, even merge them. Here's how.

downloading open-source model weights
# Install Ollama — runs open models locally
$ curl -fsSL https://ollama.com/install.sh | sh
# Download LLaMA 3.1 — 70B parameters, 40GB file
$ ollama pull llama3.1:70b
pulling manifest... done
pulling 4e9568fb1e7d... 100% ████████████ 40.1 GB
# That's it. The "weights" are in that 40GB file.
# It's literally billions of floating-point numbers.
$ ollama run llama3.1:70b
>>> Model loaded. You're running AI locally.
Fine-tuning
Take existing weights, train on YOUR data (code repos, domain docs, custom formats). The model adapts its weights to your domain. Tools: Unsloth, Axolotl, HuggingFace TRL.
LoRA / QLoRA
Don't retrain all 70B params — freeze most, add small adapter layers (only ~0.1% of weights). Train THOSE. You get a tiny file (50-200MB) that modifies the base model's behavior. Fast and cheap.
Quantization
Original weights = 16-bit floats. Quantize to 4-bit = model shrinks 4x but loses ~5% quality. GGUF format. This is why you can run 70B models on a gaming PC with 24GB VRAM.
Model merging
Take two fine-tuned models and merge their weights mathematically (SLERP, TIES, DARE). Get a hybrid that combines both skills. It's like breeding AIs. Wild stuff happening on HuggingFace.

Open vs Closed: LLaMA, Mistral, DeepSeek, Qwen — you can download and run these. Claude, GPT-4, Gemini — closed weights, API-only. But you can still deeply customize closed models through system prompts, tool definitions, and configuration files.

CHAPTER 05 // FINE-TUNING
Retraining a model on your data

Fine-tuning takes a pre-trained LLM and further trains it on YOUR specific data — your codebase, your documents, your domain. The model's weights shift slightly to become an expert in your world.

▸ WHICH MODELS CAN YOU FINE-TUNE?

DeepSeek R1/V3
● FULL ACCESS
Open weights. Download & fine-tune locally. 671B MoE model, distilled versions (7B-70B) easier to train. Apache 2.0 license.
LLaMA 3.1
● FULL ACCESS
Meta's open weights. 8B, 70B, 405B variants. Most popular base for fine-tuning. Massive community & tooling support.
Mistral / Mixtral
● FULL ACCESS
Open weights. 7B, 8x22B MoE. Great efficiency. Popular for production fine-tunes. Apache license.
Qwen 2.5
● FULL ACCESS
Alibaba's open weights. 0.5B to 72B. Strong multilingual & code. Very competitive with LLaMA.
GPT-4o / GPT-4o-mini
● API FINE-TUNE
Upload JSONL via OpenAI API. They train on their servers. You get a custom model ID. No weight access — you rent the result.
Claude (Anthropic)
● API FINE-TUNE
Available for enterprise. You send training data to Anthropic, they fine-tune on their infra. Custom model accessible via API. No weight download.
Gemini (Google)
● API FINE-TUNE
Google AI Studio or Vertex AI. Upload data, tune via API. Result hosted on Google Cloud. You don't get the weights.
Grok (xAI)
● LIMITED
Grok-1 weights released (314B MoE). Fine-tuning possible but impractical — requires massive hardware. No API fine-tune yet.

▸ THE FINE-TUNING PROCESS — STEP BY STEP

1. Prepare your dataset
Convert your data into training format: prompt/completion pairs (JSONL). For coding: input = task description, output = ideal code. Need 100-10,000+ examples depending on the task. Quality > quantity — garbage in = garbage out.
2. Choose your method
Full fine-tune — retrain all weights. Best quality, needs 4-8x A100 GPUs ($2-10K+ per run).
LoRA — freeze base, train small adapters. 10-100x cheaper. 95% of full quality. Most people use this.
QLoRA — LoRA on a quantized (4-bit) model. Runs on a single GPU. ~90% of full quality.
3. Train
Feed examples through the model. Calculate loss (how wrong was the prediction?). Adjust weights using backpropagation. Repeat for 1-5 epochs. Monitor for overfitting — if the model memorizes your data instead of learning patterns, it'll fail on new inputs.
4. Evaluate
Test on held-out data the model never saw during training. Compare outputs against your ground truth. Metrics: perplexity, BLEU, human eval. If quality is bad → more/better data, adjust hyperparameters, try different base model.
5. Deploy
Export weights (or LoRA adapter). Quantize to 4-bit for production. Serve via vLLM, TGI, Ollama, or upload to HuggingFace. For API fine-tunes (OpenAI/Anthropic) — you just get a model ID to use in API calls.

▸ WHEN TO FINE-TUNE vs WHEN NOT TO

Fine-tune when...
• You need a specific output format consistently
• Domain-specific language (legal, medical, Latvian accounting)
• You have 500+ high-quality examples
• Prompt engineering alone isn't hitting your quality bar
• You need lower latency (smaller fine-tuned model > big general model)
• You want to reduce per-request costs at scale
Don't fine-tune when...
• A good system prompt + few-shot examples solve it
• You have <100 examples (will overfit)
• The task requires broad general knowledge
• RAG (retrieval) would work better (dynamic data)
• You need to update knowledge frequently (retrain is slow)
• You just want the model to "know" your docs (use RAG instead)
CHAPTER 06 // THE AGENT
How LLMs become coding agents

The model alone just predicts text. Wrap it in a tool-calling loop with file access, shell commands, and git — now it's an autonomous coding agent. This is how Cursor, Copilot, Claude Code, Windsurf, and every AI IDE works.

▸ CLICK EACH STEP TO SEE THE DETAILS

STEP 01
You give a task
You type: "Fix the failing auth tests"
The agent wraps your message in a system prompt containing:
• Project context (file tree, recent changes, docs)
• Available tools (read files, edit, run commands, search)
• Rules & constraints (from config files like CLAUDE.md, .cursorrules, etc.)
• Conversation history

This assembled prompt can be 50,000-200,000 tokens long.
STEP 02
LLM reasons & picks a tool
The model doesn't output text — it outputs a structured tool call.
// The LLM returns JSON like this: { "tool": "bash", "args": { "command": "npm test -- src/auth/ --no-coverage" }, "reasoning": "Need to see which tests fail first" }
// This is called "function calling" or "tool use". // The model was trained to output structured JSON tool calls // instead of plain text when tools are available.
STEP 03
Agent executes the tool
The host app runs the command on YOUR machine and captures the output.
// The agent (Cursor, Claude Code, Copilot, etc.) does: 1. Parse the tool call JSON 2. Check permissions (should this be allowed?) 3. Execute: spawn("npm test -- src/auth/") 4. Capture stdout + stderr 5. Feed result back to the LLM as a new message
// The LLM never touches your filesystem directly. // The agent app is the middleware.
STEP 04
LLM observes & decides next action
Sees test output, reads relevant files, makes edits, runs tests again.
// The loop: while (!done) { response = LLM(system_prompt + conversation + tool_results) if (response.has_tool_call) { result = execute_tool(response.tool_call) conversation.append(result) } else { // Model returned plain text = it's done show_to_user(response.text) done = true } }
// This "reason → act → observe" loop IS the agent. // Every AI coding tool uses this exact pattern.
STEP 05
Task complete
Tests pass. The model returns a text summary instead of another tool call.
A single "fix the auth tests" task might involve:
• 1 bash call (run tests) • 2 file reads (source + test file) • 1 file edit (fix the bug) • 1 bash call (re-run tests) • 1 git commit
= 6 tool calls, 6 LLM roundtrips, ~30 seconds total.
// Complex tasks can be 50-100+ tool calls.

▸ THE TOOLS EVERY AI CODING AGENT HAS

Read / Search Files
Read specific files or search the codebase using grep/ripgrep/AST parsing. The model requests EXACTLY what it needs — it doesn't load your whole repo into context.
Edit Files
Targeted diffs, not full file rewrites. The model specifies "replace lines 42-48 with..." rather than regenerating the whole file. This keeps edits precise and reviewable.
Execute Commands
Run tests, install packages, build the project, start servers. Shell access is what makes it an agent vs. a chatbot. The model sees stdout/stderr and reacts.
Sub-agents
Spawn parallel workers for complex tasks. "Search for all auth-related files" and "check test coverage" can run simultaneously. This is how agents handle large codebases efficiently.
CHAPTER 07 // CONTEXT
The context window problem

An LLM can only "see" a fixed amount of text at once. This limit — the context window — is the single biggest constraint in AI coding. Everything else is engineering around it.

▸ HOW AI CODING TOOLS DEAL WITH THIS

Smart Retrieval
Don't load whole files — use AST parsing, symbol search, and grep to pull ONLY the relevant functions and classes. Tree-sitter parses code structure so the agent knows what to grab.
Context Compaction
When the window fills up, older messages get summarized. "You edited auth.ts lines 42-48 to fix UTC comparison" instead of keeping the full file diff in context.
Embeddings + RAG
Index the codebase into a vector database. When you ask about auth, retrieve the top-k most relevant code chunks by semantic similarity. Cursor and Copilot do this aggressively.
Multi-Agent
Spawn sub-agents with fresh context windows for sub-tasks. Each gets a focused slice of the problem. Results get compressed and sent back to the main agent. Parallelism + isolation.

▸ AI CODING TOOLS COMPARED

Cursor / Windsurf
IDE-based. Deep editor integration. Sees your open tabs, cursor position, selections. Good for inline completions + chat. Uses RAG to index your project. Multiple model choices.
GitHub Copilot
VS Code extension + CLI. Inline completions (fast, small model) + chat agent (larger model). Workspace indexing. Tight GitHub integration for PRs and issues.
Claude Code
Terminal-based CLI agent. No IDE needed. Full filesystem + shell access. Spawns sub-agents. MCP protocol for external tools. Raw power for large refactors and complex tasks.
Aider / Open Source
Open-source CLI agents. Works with any LLM (local or API). Full transparency — you can read the source. Community-driven. Good for learning how agents actually work.

Under the hood, they ALL use the same pattern: assemble context → call LLM → execute tool → feed result back → repeat. The differences are in UX, context management, and which models they support.
